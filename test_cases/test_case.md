# Тест-кейсы для проверки сайта Позитроника

## Общая информация
Данный набор тест-кейсов предназначен для проверки функциональности формы обратной связи на сайте [Позитроника](https://positronica.ru/support/). Тесты включают в себя проверку валидации полей, корректности отправки данных и взаимодействия с элементами интерфейса.

**Важно:**  
Этот документ служит исходным материалом для ручного тестирования и предварительной автоматизации. В текущем виде описания являются высокоуровневыми и требуют доработки для полноценного автоматизированного тестинга. Для ручного тестирования рекомендуется использовать эти кейсы как основу, дополняя их конкретными деталями, ожидаемыми результатами и сценариями проверки.

---

## Предварительные условия
- Установлен драйвер Chrome и настроена среда для автоматизации (при необходимости).
- Браузер открыт, сайт доступен, форма обратной связи находится на видимом месте.
- Перед началом тестирования рекомендуется очистить кэш и куки для исключения влияния предыдущих сессий.

### Тест-кейс 1: Проверка валидации и функциональности сайта Позитроника

**ID:** TC001  
**Название:** Проверка валидации полей ввода и функциональности кнопок сайта Позитроника

**Предварительные условия:**  
- Страница формы загружена полностью.  
- Все поля пустые.

**Проверяемые значения:**   
1. Найти раздел с формой обратной связи.  
2. В поле "Город" открыть выпадающий список и выбрать один из доступных вариантов (например, "Москва").  
3. В поле "Email" ввести: `ivanov@mail.ru`  
4. В поле "Телефон" ввести: `+7 987 654 32 10`  
5. В поле "Имя" ввести: `Иван`  
6. В поле "Сообщение" ввести: `Тестовое сообщение`  
7. Поставить галочку напротив "Я ознакомлен и согласен с условиями Политики конфиденциальности"  
8. Нажать кнопку "Отправить".

**Ожидаемый результат:**  
- Все поля отображают введённые данные.  
- При заполнении валидных данных кнопка "Отправить" становится активной.  
- После нажатия происходит отправка данных.  
- Во время отправки появляется индикатор загрузки, кнопка блокируется, чтобы предотвратить повторные отправки.

### Тест-кейс №3: Проверка валидного поля Email

**ID:** TC003
**Название:** Проверка валидации поля Email

### Предварительные условия
- Страница формы загружена
- Поле "Email "пустое

****Проверяемые положительные значения:****
- `test@example.com` — ввод валидного email.
- `test123@example.com` — ввод валидного email с цифрами.
- `test @example.com` - пробелы автоматически удаляются, ввод валидного email.
- `firsttest.lasttest@domain.co` — ввод многоуровневого валидного email.
- `TEST@EXAMPLE.COM` — ввод валидного email в верхнем регистре.
- `testtesttesttesttesttesttesttesttesttesttesttesttesttesttesttestttesttesttesttesttest
testtesttestesttesttest@example.com` - максимально допустимая длина, ввод валидного email.
- `t@example.com` - минимально допустимая длина, ввод валидного email.
- `test@example.co` - сжатие домена до двух символов, email принимается.

****Проверяемые отрицательные значения:****
- `test@example.c` - сжатие домена до одного символа, сообщение "Неверный формат Email".
- `test@example` - отсутствие домена, сообщение "Неверный формат Email".
- `?test+@,example!.com` - ввод email с недопустимыми символами, сообщение "Неверный формат Email".
- `мейл` - сообщение "Введите Email латинскими буквами".


**Ожидаемый результат:**
- При вводе валидного email адреса сообщение об ошибке отсутствует.
- При вводе невалидного email-адреса, система отображает сообщение об ошибке, указывающее на некорректный формат email.
- Ввести валидное значение в поле email (`test@example.com`), сообщение об ошибке пропадает.

### Тест-кейс 10: Проверка выбора категории обращения

**ID:** TC010  
**Название:** Проверка возможности выбора одной категории обращения из четырёх на странице поддержки

**Предварительные условия:**  
- Страница формы открыта в браузере.

**Проверяемые значения:**
1. Перейти к разделу выбора категории обращения.  
2. Нажать на переключатель рядом с категорией "Не могу оформить заказ".  
3. Проверить, что эта категория выбрана (галочка установлена).  
4. Кликнуть на другую категорию, "Проблема с доставкой".  
5. Проверить, что выбранная ранее категория снимается, а новая — установлена.  
6. Повторить для каждой из остальных двух категорий: "Не подошло/Не хватает комплектующих/Не работает" и "Вопрос/Пожелание", выбирая их по очереди и проверяя, что выбор переключается корректно.

**Ожидаемый результат:**  
- Можно выбрать только одну категорию обращения за раз.  
- При выборе новой категории, ранее выбранная снимается автоматически.  
- После выбора любой категории, она отмечена галочкой, подтверждая выбор.

## Применяемые техники тестирования
1. Граничные значения (Boundary Value Testing): Проверка на границах допустимых значений для длины вводимых сообщений.  
2. UI-тестирование (UI Testing): Проверка доступности и корректности элементов интерфейса: взаимодействие с кнопками и полями, отображение подсказок и подсветок, состояние элементов (активные/неактивные).  
3. Интеграционное тестирование (Integration Testing): Проверка взаимодействия с базой данных при отправке данных.  
4. Тестирование на основе требований (Requirement-based Testing): Проверка соответствия реализации заявленным требованиям к полям ввода и кнопкам.

## Дополнения и рекомендации по автоматизации

- Для автоматизации тестов рекомендуется добавлять проверки наличия подсказок или ошибок в DOM, их видимости, а также состояния кнопки "Отправить" (активна/неактивна).  
- В тестах стоит предусматривать проверки успешной отправки (например, наличие сообщения об успехе или редиректа).  
- Следует реализовать сценарии тестирования граничных значений для длины сообщений, форматов данных, а также тестирование реакции системы на поведенческие сценарии (например, повторная отправка, отмена).
