# Тест-кейсы для проверки сайта Позитроника

## Общая информация
Данный набор тест-кейсов предназначен для проверки функциональности формы обратной связи на сайте [Позитроника](https://positronica.ru/support/). Тесты включают в себя проверку валидации полей, корректности отправки данных и взаимодействия с элементами интерфейса.

**Важно:**  
Этот документ служит исходным материалом для ручного тестирования и предварительной автоматизации. В текущем виде описания являются высокоуровневыми и требуют доработки для полноценного автоматизированного тестинга. Для ручного тестирования рекомендуется использовать эти кейсы как основу, дополняя их конкретными деталями, ожидаемыми результатами и сценариями проверки.

---

## Предварительные условия
- Установлен драйвер Chrome и настроена среда для автоматизации (при необходимости).
- Браузер открыт, сайт доступен, форма обратной связи находится на видимом месте.
- Перед началом тестирования рекомендуется очистить кэш и куки для исключения влияния предыдущих сессий.

## Тест-кейсы

### Тест-кейс 1: Проверка валидации полей ввода и функциональности кнопок сайта Позитроника

**ID:** TC001  
**Название:** Проверка валидации полей ввода и функциональности кнопок сайта Позитроника

**Цель:**  
Подтвердить, что все поля формы корректно валидируются и кнопка "Отправить" функционирует в соответствии с требованиями.

**Предварительные условия:**  
- Страница формы загружена полностью.  
- Все поля пустые.

**Шаги:**  
1. Перейти на [https://positronica.ru/support/](https://positronica.ru/support/)  
2. Найти раздел с формой обратной связи.  
3. В поле "Город" открыть выпадающий список и выбрать один из доступных вариантов (например, "Москва").  
4. В поле "Email" ввести: `ivanov@mail.ru`  
5. В поле "Телефон" ввести: `+7 987 654 32 10`  
6. В поле "Имя" ввести: `Иван Иванов`  
7. В поле "Сообщение" ввести: `Тестовое сообщение`  
8. Поставить галочку напротив "Я ознакомлен и согласен с условиями Политики конфиденциальности"  
9. Нажать кнопку "Отправить".

**Ожидаемые результаты:**  
- Все поля отображают введённые данные.  
- При заполнении валидных данных кнопка "Отправить" становится активной.  
- После нажатия происходит отправка данных (может отображаться сообщение об успешной отправке или редирект).  
- Во время отправки появляется индикатор загрузки или кнопка блокируется, чтобы предотвратить повторные отправки.

### Тест-кейс 2: Проверка валидации полей при неправильном вводе данных

**ID:** TC002  
**Название:** Проверка валидации полей при неправильном вводе данных

**Цель:**  
Проверить, что система корректно реагирует на некорректные или неполные данные и отображает подсказки.

**Предварительные условия:**  
- Страница формы загружена, поля пустые.

**Шаги:**  
1. Перейти на [https://positronica.ru/support/](https://positronica.ru/support/)  
2. В поле "Email" ввести некорректный email: `мейл`
3. В поле "Email" ввести неполный адрес email, например: `ivanov@mail`
4. В поле "Email" ввести несуществующий адрес email, например: `fake@mail.ru`
5. В поле "Телефон" ввести неполный числовой набор
6. В поле "Телефон" попытаться ввести буквенный набор
7. В поле "Имя" ввести имя латинскими буквами
8. В поле "Имя" ввести имя длиной 1 символ
9. В поле "Сообщение" ввести короткое сообщение: `Три`

**Ожидаемые результаты:**  
- Под полями отображаются подсказки или подсветки красным, указывающие на необходимость корректного ввода (например, "Неверный формат Email").  
- Если форма позволяет отправку с некорректными данными, то при отправке появляется сообщение об ошибке.  
- Поля остаются выделенными, пока пользователь не исправит ошибки.  
- Кнопка "Отправить" становится неактивной, если обязательные поля заполнены неправильно.

### Тест-кейс 3: Проверка обязательных полей при попытке отправки без заполнения

**ID:** TC003  
**Название:** Проверка обязательных полей при отправке формы без заполнения

**Цель:**  
Подтвердить, что система не допускает отправку формы без заполнения обязательных полей и правильно отображает подсказки.

**Предварительные условия:**  
- Страница формы открыта, поля пустые.

**Шаги:**  
1. Не заполнять поля "Email", "Телефон", "Город", "Сообщение".  
2. Не ставить галочку о согласии с политикой.  
3. Нажать кнопку "Отправить".

**Ожидаемые результаты:**  
- Под каждым обязательным полем отображается подсказка или подсветка с сообщением о необходимости заполнения (например, "Это поле обязательно").  
- Кнопка "Отправить" либо неактивна, либо при попытке нажать — ничего не происходит.  
- После заполнения всех обязательных полей и установки галочки, кнопка становится активной и форма отправляется успешно.

### Тест-кейс 4: Проверка выбора категории обращения

**ID:** TC004  
**Название:** Проверка выбора одной категории обращения из четырёх на странице поддержки

**Цель:**  
Убедиться, что пользователь может выбрать только одну категорию обращения из предложенных четырёх и при переключении с одной категории на другую предыдущий выбор снимается.

**Предварительные условия:**  
- Страница формы открыта в браузере.

**Шаги:**  
1. Перейти к разделу выбора категории обращения.  
2. Нажать на галочку или переключатель рядом с категорией "Не могу оформить заказ".  
3. Проверить, что эта категория выбрана (галочка установлена).  
4. Кликнуть на другую категорию, например, "Проблема с доставкой".  
5. Проверить, что выбранная ранее категория снимается, а новая — установлена.  
6. Повторить для каждой из остальных двух категорий: "Не подошло/Не хватает комплектующих/Не работает" и "Вопрос/Пожелание", выбирая их по очереди и проверяя, что выбор переключается корректно.

**Ожидаемые результаты:**  
- Можно выбрать только одну категорию обращения за раз.  
- При выборе новой категории ранее выбранная снимается автоматически.  
- После выбора любой категории, она подсвечена или отмечена галочкой, подтверждая выбор.

**Ожидаемые результаты:**  
- После клика происходит переход на указанную страницу.  
- URL в адресной строке соответствует https://positronica.ru/pretenzii-po-zakazam-s-markepleysov/

## Применяемые техники тестирования
1. Граничные значения (Boundary Value Testing): Проверка на границах допустимых значений для длины вводимых сообщений.  
2. UI-тестирование (UI Testing): Проверка доступности и корректности элементов интерфейса: взаимодействие с кнопками и полями, отображение подсказок и подсветок, состояние элементов (активные/неактивные).  
3. Интеграционное тестирование (Integration Testing): Проверка взаимодействия с базой данных при отправке данных.  
4. Тестирование на основе требований (Requirement-based Testing): Проверка соответствия реализации заявленным требованиям к полям ввода и кнопкам.

## Дополнения и рекомендации по автоматизации

- Для автоматизации тестов рекомендуется добавлять проверки наличия подсказок или ошибок в DOM, их видимости, а также состояния кнопки "Отправить" (активна/неактивна).  
- В тестах стоит предусматривать проверки успешной отправки (например, наличие сообщения об успехе или редиректа).  
- Следует реализовать сценарии тестирования граничных значений для длины сообщений, форматов данных, а также тестирование реакции системы на поведенческие сценарии (например, повторная отправка, отмена).
